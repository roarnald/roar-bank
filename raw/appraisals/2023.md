| Technical Achievements (I left out several of the day to day features and only wrote prominent and impactful changes) ***gitlab-pipeline*** \- As we migrated Jenkins, we took the chance to refactor part of the CI/CD process, instead of git polling in the past, we decided to use gitlab runner to curl jenkins build URL, making it easier to identify which Jenkins each codebase is linked to, as one codebase can be used for several project/instances.  **UIUX Revamp** \- I spent most of my time during the first quarter of the year researching, fixing, and automating a lot of manual work that were present in several UIUX Repositories. For example, game rules were manually built in a developer’s local machine and uploaded directly to all environments. Building locally was a common workflow for UIUX Repositories. They also only run on a single master branch most of the time, making work very manual and hyper reliant on the human remembering the build and deploy process (irk\!). It was so manual that the team had multiple Word documents listing the different steps for each project. The following repositories had the biggest changes: **CASH Desktop Franchise** \- The biggest holdback for this project is the requirement from APPSUPPORT Team to be able to make Frontend changes on their own, e.g. changing phone numbers, etc. Hence, firstly, the build files were built locally and committed. Secondly, there will be random commits to these build files and they have to maintain an excel sheet for developers to guess which source code will match the modified build file, making it incredibly manual.  I added an automated script to run a gif diff on SIT to identify any commits that were added but missing from SIT, then rebased SIT. So, when trying to commit an out of sync build file, git will be able to identify the difference early, instead of having developers check for updates on an excel sheet. **CDN-Shine** \- A JS CDN file that is used for all Desktop Franchise Templates. Before this year, they only had 1 x production file serving for all environments. We would often find ourselves wrongly deploying features that were not ready to production as there was only one single file. This year we split the build into SIT, UAT and production. *Setting up runners and link to (1)* **Forking of H5 Games Repo** \- They used to use the same codebase for Cash and API H5 Games, by using different branches, e.g. production and production-api. Split them into different repositories. **IPA Dashboard** \- This was an internal helper dashboard for APP Team to manage metadata. However, it was written with several legacy dependencies, and  is largely slow. I purged all unused packages, rewrote legacy packages and improved its performance score from 60+ to 99\. **DYH Desktop, DYH-VN Desktop** \- These membersite desktop projects are efforts from Tsai Yu to split workload for the UIUX Team and are the golden examples that I will always bring up to back my reasoning to dissolve the team. Upon inspection by any senior React Team dev, it is plainly obvious that the developer that set up the project had little to no understanding on how several tools worked. Also, half of the dependencies were legacy **AND** unused. Before merger, UIUX developers would also spend weeks trying to “optimize” something that would take React Team developers an hour or so to do up, and they would spend those weeks coming out with little to no results. It was a huge waste of resources and more added stress for us as we scale as a company and team. Examples include:  Header bar was rendered once for every single page, instead of reusing. It was also specially stated that ***no handover was allowed***. Also, our expectations could be better managed as well, so we can, as Team Leads, assign the appropriate resource, instead of hard swallowing everything at our own expense. **DYH Revamp** \- A rewrite of this project is much needed. As mentioned, this repository was previously written by the then UIUX Team. Several of its code is dated and its internal dependencies (like subpages) were hard to pinpoint, making it extremely hard to maintain and debug. I also took the chance and did a sharing during Frontend Sharing on the experiments and improvements that I made in this iteration, including but not exclusive of: **@frontend/service** \- One of the biggest decisions when writing react apps is choosing the state management system. First written by Tsai Yu, our frontend service has evolved over the years. Last year, I introduced dynamic key watching to allow greater flexibility in optimization while using services. This year, I contributed to its third iteration, which included one of the takeaways from Frontend Sharing, making it slightly faster in speed. **\<dialog\>** \- As we are starting to use monorepo to write projects, there was a need to use solutions that had as few dependencies as possible. Dialog is a modern built in way to write pop ups for web browsers, achieving wide browser support in 2022\. We used this to build shared components that could be used anywhere, as lightweight as it can be, which serves nicely into introducing newer technologies and architectures, like, **NextJS** \- This “Server Side Rendering Framework” has been the talk within our team since I first joined 4 years ago. In 2022, I completed a feature to migrate DYHVN into a single SSR Application but our DevOPs team was too unfamiliar with the hosting of the application, and Vercel Enterprise didn’t make financial sense. This year, I dug deep into the NextJS documentation and found that one of the ways NextJS was designed is to code split the traditional React App by rendering during build time. This year was also the release of React 18 \+ Next 13 (Long controversial story that I could talk for hours). In a nutshell, React 18 was designed purely for Vercel to be used in NextJS, hence there were many features that were tightly coupled together. By leveraging this, I pulled off a build time rendered project for the **mobile** and **subpages** subpackage in DYH Revamp. Unfortunately, this project was put on hiatus prematurely until further notice after being 80% complete. **MDX** \- This tool was a godsend. While setting up this monorepo, I noticed one of the top tools being mdx. Curious on its popularity, I dug into their documentation and was blown away. This tool alone has skyrocketed our productivity in making long form content by up to 90%. In the past, when setting up long wordy pages, the UIUX team would have to copy each line and wrap them in the correct HTML Tag. With MDX, all I had to do was copy the word file given in the requirements and add the appropriate headers in markdown. *I also enforced the use of MDX in another project which I was not involved in \- GT Credit OG*, *and got positive feedback on its insane ease of development.* **Complete rewrite of all internal project dependencies** \- Typically, our project has several reliance on old legacy repositories like (yes each of this sub point belongs to different repository): Yu E Bao Graph Yu E Bao FAQ Spin Wheel Lucky Draw DYH Subpages \- (签到送, 常见问题, etc.) Puzzle Swipe Captcha Payment Page I rewrote each and every single one of them while ensuring that they were fast and ergonomic to use, to be used by Desktop, Mobile Web and also APP. Hence, the workload for Web is almost **tripled** that of APP Team’s. **New Relic & Monitoring CDN** \- Frontend monitoring was on our horizon for a while. We were about to really dive into researching tools like Sentry and DataDog but was met with a timely New Relic purchase from the company. After much thought, we designed a separate JS file that cached the New Relic official script, and added our own filters to reduce data usage and saving cost incurred. **Adhoc Support for UIUX Requests** \- With the absence of a UIUX Team, I had to find solutions to many of the ongoing adhoc request like, Maintenance Pages, Product Landing Pages and external projects. **Orion+** \- Our Frontend Frankenstein. This was never conceptualized to be anything like what it is today. This year on top of the nitty gritties, a few notable changes are, Integrating Techfi Calendar and Migrating Passwords to Vault. **Svelte** \- We finally did it. This year I saw an opportunity to push for a very (and I mean, *very*) trendy framework in the open sourced community, Svelte JS. Our reliance on React has been alarming for a couple of years. Tsai Yu had a motivating vision, that *we should strive to be Javascript developers, and not React Developers*. Today, I am proud to say that we finally did it. Despite this just being just a small step, it is a significant one. Although this project was done by a fellow developer, I took the initiative to dive deep into the documentation, so that I could code review and steer the codebase in the proper direction, while we navigate this new framework with inexperience.
