# Context

During code reviews, a recurring tension arose: reviewers would ask leading questions to help developers learn, but developers with urgent fixes felt mounting pressure from wanting to push a fix quickly while trying to absorb a concept. Two incidents of high tension were noticed by Ronald.

# Problem

A mismatch in emotions existed between code reviewers (trying to coach) and the developer requesting review (facing urgency). Reviewers could not tell if a request was urgent, so they defaulted to teaching mode, which felt obstructive during genuine emergencies.

# Actions

- Took initiative to **individually interview each affected developer** to understand the root cause of the tension.
- Discussed findings with the Frontend Leads (Tsai Yu, Joanne).
- **Proposed and implemented Urgency Tagging**: developers label Merge Requests with an `[Urgent]` tag when genuinely time-pressured. When tagged, reviewers must provide an immediate fix; coaching is deferred.
- Strongly encouraged against overusing the tag to preserve the learning culture.
- Conducted a **follow-up feedback session 2 months after implementation** — most developers reported feeling more at ease.

# Impact

- Resolved recurring source of friction in the team workflow.
- Team morale improved; developers felt leads were listening and responsive.
- The `Urgent` tag was rarely used — which was the intended design. Its mere existence as a safety valve provided peace of mind.
- Demonstrated systems thinking: identifying root cause (emotional mismatch), not just the symptom (tension).
